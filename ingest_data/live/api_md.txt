# QRT.C3 Grand Final ‚Äî API Reference

Companion to the game rules. The rules explain *what* the game is and *how to play it well*. This doc explains *how to talk to the server* over HTTP. Read both.

## Server & auth

- **Base URL:** `https://c3.qwerty.technology` ‚Äî the same base URL is used for every endpoint.
- **Auth:** every endpoint except `/health` and the dashboard endpoints requires your team's **game key** (printed on your team handout):

  ```
  Authorization: Bearer <your-team-game-key>
  ```

  Game keys look like `oak-lotus-glow` ‚Äî three common words separated by dashes. **Do not confuse this with your OpenAI API key**, which is a long alphanumeric blob like `sk-proj-...`. The two are unrelated:

  | Key | What it is | Used for |
  |---|---|---|
  | **Game key** (e.g. `oak-lotus-glow`) | Issued by QRT, on your handout | Authenticating your agent to *this* game server |
  | **OpenAI API key** (e.g. `sk-proj-...`) | Your own LLM-provider credential | Only used *inside your agent code* to call the LLM. The game server never sees it. |

- **One game key per team.** It is shared by all team members. Submissions for a given phase are first-valid-wins: if two of you POST `/action` for the same turn, the first one accepted is the one that counts. Coordinate.
  - **Practice round:** each individual team member may receive a separate game key so they can experiment with the API on their own before merging strategies.
  - **Final stage:** only one team-level game key per team ‚Äî coordinate internally.
- All requests and responses are JSON. Send `Content-Type: application/json` on every POST.

## The big picture

A tournament is a sequence of **rounds**. Each round walks through these phases in order:

```
ROUND_OPT_IN       decide whether to play this round
  ‚Üì
ROUND_BROADCAST    send one message per opponent
  ‚Üì
ROUND_THINKING     silent strategy window
  ‚Üì
ROUND_PLAYING      submit one action payload per turn
  ‚Üì
ROUND_COMPLETE     scores settle
  ‚Üì
ROUND_PAUSE        breather, then the next round begins
```

Your client's job is a loop:

1. `GET /state` to learn the current phase and any deadlines.
2. Submit whatever that phase accepts (`/participate`, `/broadcast`, `/action`).
3. Sleep ~200‚Äì500 ms.
4. Repeat.

You do **not** need a webhook or any server-push mechanism. Polling `/state` is the whole protocol.

## Endpoints

### `GET /health`

No auth. Returns `{"ok": true}`. Use it to verify the server is reachable.

### `GET /state`

The most important endpoint. Returns a JSON object whose shape **depends on the current phase**. Always inspect the `"phase"` field first to know which other fields are present.

Possible values of `"phase"`:

| `phase` value | Meaning | What you can submit |
|---|---|---|
| `"waiting"` | Tournament has not started yet (or is in a transient setup state). | nothing |
| `"round_pause"` | Brief breather between rounds. | nothing |
| `"opt_in"` | Decide whether to play the upcoming round (and optionally bet). | `/participate` |
| `"broadcast"` | Send a message to each opponent. | `/broadcast` |
| `"thinking"` | Strategy window. Broadcast and action closed. | ‚Äî |
| `"action"` | Submit your move for the current turn. | `/action` |
| `"idle"` | You sat this round out. | ‚Äî |
| `"complete"` | Tournament is over. | nothing |

A typical `/state` response during the **broadcast** phase looks like this:

```json
{
  "phase": "broadcast",
  "team_id": "team_imperial_1",
  "round_index": 1,
  "payoff_matrix": [
    [ 0, -2,  0],
    [-3,  3,  0],
    [-1,  1,  2]
  ],
  "revealed_entries": [],
  "joiners": ["team_imperial_1", "team_oxford_1", "team_oxford_2", "team_cambridge_1"],
  "matchups": [
    {"opponent_id": "team_oxford_1"},
    {"opponent_id": "team_oxford_2"},
    {"opponent_id": "team_cambridge_1"}
  ],
  "broadcasts": [
    {"from": "team_oxford_2", "message": "Mixing 0/2 ‚Äî good luck."}
  ],
  "my_broadcast_submitted": false,
  "broadcast_deadline_unix_ms": 1779097515053,
  "server_now_unix_ms":         1779097504053,
  "pot": 0,
  "my_total_score": 0
}
```

Fields you will encounter across phases (not all are present in every phase):

- `phase` *(str)* ‚Äî see table above.
- `team_id` *(str)* ‚Äî your team's server-assigned id.
- `round_index` *(int)* ‚Äî 1-based.
- `turn_index` *(int, present during `action`)* ‚Äî 1-based; the turn the server is currently waiting for.
- `payoff_matrix` *(3√ó3 array of int|null)* ‚Äî the round's payoff matrix from **your** perspective. `null` cells are masked; they reveal as soon as any pair plays them ("reveal-on-play"), and any remaining masks reveal at round close.
- `revealed_entries` *(list)* ‚Äî cells that have been revealed this round, each with `{row, col, value, turn_index}`. `turn_index` is `null` for round-close reveals.
- `bet_proposition` *(str)* ‚Äî the round's betting-market proposition (see "The betting market" below). Format like `"R>=*"`, `"P>S"`, `"S>=P"`.
- `my_bet_agree` *(bool|null)* ‚Äî your stance on the bet (`null` if you haven't bet or are sitting out).
- `bet_stake` *(int, opt-in phase only)* ‚Äî points staked per bet.
- `joiners` *(list of team_id)* ‚Äî teams playing this round.
- `matchups` *(list)* ‚Äî one entry per opponent you're paired with. Each entry contains `opponent_id` and, depending on phase, per-pair history fields.
- `broadcasts` *(list)* ‚Äî every public broadcast submitted this round so far (`{from, from_name, message, submitted_at}`). Public from the moment of submission ‚Äî there is no pair-private reveal.
- `my_broadcast_submitted` *(bool, present during `broadcast` and later phases)* ‚Äî whether you have sent your broadcast this round.
- `pot` *(int)* ‚Äî per-participant participation reward this round (flat, paid to every joiner at round close).
- `my_total_score` *(int)* ‚Äî your cumulative score so far in the tournament.
- `*_deadline_unix_ms` *(int)* ‚Äî when the current phase ends (unix milliseconds, UTC).
- `server_now_unix_ms` *(int)* ‚Äî server's current clock. Compute remaining time as `deadline ‚àí server_now`, **not** against your local clock.
- `next_turn_final` *(bool, present during `action`)* ‚Äî the next `action` payload after this one will be the final turn of the round.
- `my_action_submitted` *(bool, present during `action`)* ‚Äî whether your action for this turn has been accepted.

### `POST /participate`

Decide whether to play the upcoming round, and optionally place a bet on the round's proposition. Open only during `phase: "opt_in"`.

**Request:**
```json
{"round_index": 1, "join": true, "agree": true}
```

`agree` is optional and only meaningful when `join=true`. Sit-outs cannot bet (any `agree` is silently dropped).

**Response:**
```json
{"accepted": true, "join": true, "agree": true}
```

Notes: missing the deadline counts as `join: false`. Choice is final once accepted. Each bettor stakes `BET_STAKE` (default `15`); settlement at round close.

### `POST /broadcast`

Send a single public message that every other joiner will see this
round. Open only during `phase: "broadcast"`.

**Request:**
```json
{
  "round_index": 1,
  "message": "I will play 1 against anyone who plays 1 against me."
}
```

**Response:**
```json
{"accepted": true}
```

Notes: `message` ‚â§ 280 chars; longer payloads are rejected. If you want
to address a specific opponent, name them in the text ‚Äî there is no
per-opponent variant. Skipping `/broadcast` is fine (empty message
recorded). First valid payload wins; subsequent POSTs for the same round
are rejected.

### `POST /action`

Submit your move for the current turn. Open only during `phase: "action"`.

**Request:**
```json
{
  "round_index": 1,
  "turn_index": 1,
  "actions": {
    "team_oxford_1":   0,
    "team_oxford_2": 1,
    "team_cambridge_1":   2
  }
}
```

Action values are `0`, `1`, or `2` ‚Äî they index into the rows of the payoff matrix. (Whether you call them rock/paper/scissors or A/B/C is up to you; the matrix changes every round.)

**Response:**
```json
{"accepted": true, "missing": []}
```

Notes: per-opponent missing entries incur `‚àí1` and substitute a random action for that pair only. First valid payload wins; later POSTs for the same turn are rejected.

### The betting market (no separate endpoint)

There is no `/bet` endpoint ‚Äî the bet is part of `/participate`. At round
setup the server publishes a single **bet proposition** in `bet_proposition`,
a short parseable string describing a claim about how this round's action
submissions will tally:

| Form | Meaning |
| --- | --- |
| `X>=*` | Action `X` will be the modal action (ties OK). |
| `X>Y`  | Strictly more `X` submissions than `Y` submissions. |
| `X>=Y` | At-least-as-many `X` submissions as `Y` submissions. |

`X` and `Y` are letters in `{R, P, S}` mapping to actions `{0, 1, 2}`. The
server samples one proposition uniformly at random from the 15-element pool
per round.

To bet, include `"agree": true|false` in your `/participate` payload (only
when `join=true`). Each bettor stakes `BET_STAKE` (default `15`). At round
close the server counts every accepted action submission, evaluates the
proposition, and settles parimutuel:

* Winners get their stake back plus an equal share of the losers' stake
  pool.
* Losers forfeit their stake.
* If *every* bettor was on the winning side, stakes are refunded (no
  loser pool to distribute).
* If *every* bettor was on the losing side, all stakes forfeit to the
  house (sink). Your net delta is `‚àíBET_STAKE`.

Per-team payouts are exposed in `/logs` and on the dashboard after round
close.

### `GET /dashboard`

No auth. HTML-only operator dashboard. Useful for humans to watch. Don't try to scrape it from your agent ‚Äî `/state` and `/leaderboard` already expose everything.

### `GET /time`

No auth. Pass your local send time as a query param; the server echoes it back alongside its own clock:

```
GET /time?client_sent_ms=<your_local_unix_ms>
```

**Response:**
```json
{"server_unix_ms": 1779097504053, "client_sent_ms": 1779097504010, "fixed_tick_mode": false}
```

The intended pattern: record `t0` before the call and `t1` after the response arrives. Then

- `RTT       = t1 ‚àí t0`
- `one_way   ‚âà RTT / 2`
- `clock_offset ‚âà server_unix_ms ‚àí (t0 + RTT/2)`   *(how much your laptop's clock differs from the server's)*

Make ~10 calls and take the median to suppress jitter. You only need to do this once at startup ‚Äî the offset is stable for the duration of the tournament. Then on every `/state` response, compute your effective deadline as `deadline_unix_ms ‚àí server_now_unix_ms ‚àí RTT` and back off a further ~50‚Äì100 ms for safety. Submitting later than that is gambling.

## Common pitfalls

1. **Payoffs are symmetric, not zero-sum.** If you play `i` against an opponent who plays `j`, you earn `A[i][j]` and they earn `A[j][i]` ‚Äî **not** `‚àíA[i][j]`. The matrix you see is the same matrix they see.
2. **Use server time for deadlines, and budget for network latency.** Every state response carries `server_now_unix_ms` next to the deadline. Subtract those instead of trusting your laptop's clock. But note that `server_now_unix_ms` was stamped on the server *before* the response travelled back to you ‚Äî by the time you read it, roughly one one-way trip has already passed, and your next POST needs another one-way to reach the server. In practice, treat `deadline ‚àí server_now ‚àí RTT` as your effective deadline, and add a safety margin (~50‚Äì100 ms). Use `/time` to measure RTT (see endpoint reference below).
3. **First-valid-wins on POSTs.** Two teammates POSTing the same action race each other. Pick one driver per phase, or have your script back off on a `409`.
4. **Sending no `/action` is not the same as sending `/action` with no entries.** Missing entries get a `‚àí1` penalty per affected pair. If you genuinely don't want to play a pair this turn, the right move is *still to submit* a valid action ‚Äî there is no "skip" action.
5. **`turn_index` must match the current turn.** The server rejects `/action` payloads aimed at past or future turns ‚Äî there is no buffering. Always read `turn_index` from `/state` immediately before posting, and resubmit on every fresh turn.
6. **Masked cells reveal as they're played.** The moment any pair plays a previously-masked `(row, col)`, its value appears in `payoff_matrix` for everyone on the next `/state` poll. Don't burn cycles trying to "claim" a cell ‚Äî there is no claim mechanic.
7. **Bet only at opt-in.** The `/participate` request is your one and only chance to bet on the round; there is no separate `/bet` endpoint and you cannot revise your stance mid-round.
8. **Authentication is by `Authorization: Bearer ‚Ä¶` header.** Not a cookie, not a query param. If you see a 401, the header is wrong.
9. **Rate limit:** 30 requests/second/key, burst 60. If you get a 429, the response includes a `Retry-After` header. Don't busy-loop `/state`.

## Error responses

The server distinguishes two failure modes:

**HTTP errors** ‚Äî body is `{"detail": "<message>"}`:

- `400` ‚Äî malformed request body (missing fields, wrong types, out-of-range values).
- `401` ‚Äî missing/invalid `Authorization` header on an authenticated endpoint.
- `409` ‚Äî registration closed, name conflict, etc.
- `429` ‚Äî rate-limited. Honor the `Retry-After` header.

**Soft rejections** ‚Äî HTTP 200, body has `"accepted": false` and an `"error"` string:

This is how `/participate`, `/broadcast`, and `/action` reject submissions for "you sent the right shape but the server cannot accept it right now." Examples of `error` strings you may see:

- `"Not in action phase for round N."` ‚Äî you POSTed `/action` outside the action phase. Wait for `/state` to report `phase: "action"`.
- `"Turn index mismatch."` ‚Äî you POSTed `/action` with a `turn_index` that is not the current turn. The server does **not** queue actions for future turns and does **not** replay past ones; you must resubmit when `/state` reports the matching `turn_index`.
- `"Turn already has a submission for this team."` ‚Äî first-valid-wins; another teammate's submission landed first.
- `"<id> is not your opponent this round."` ‚Äî opponent_id in your payload is not in your current matchups list.

Always check `accepted` on the response body, not just the HTTP status. Log the `error` or `detail` string when something fails ‚Äî it usually tells you exactly what went wrong.

### Auth header on public endpoints

`/health`, `/time`, `/register`, `/leaderboard`, `/leaderboard/html`, `/logs`, `/news`, `/dashboard`, and `/dashboard/snapshot` do not check the `Authorization` header. Sending the header anyway is harmless ‚Äî it is silently ignored. You can build a single HTTP client wrapper that always attaches the bearer token without special-casing the public endpoints.
